# Phase 3：出规划

本文件定义 `/learn-plan` Phase 3 的执行要求。此阶段合并了原 approval 和 finalize。

## 1. 目标

基于 Phase 1 的综合报告 + Phase 2 的诊断结论（如有），产出正式长期学习计划。

## 2. 两步节奏

```text
  草案 + 确认（可多轮修改）
    → 正式落盘
    → 提示进入 /learn-today
```

## 3. 草案结构

计划草案至少包含：

### 学习画像
- 学习目的、成功标准
- 当前水平（基于诊断/自报）
- 时间约束、教学偏好、练习偏好
- 非目标

### 能力指标与起点判断
- 达成目标需要掌握的核心能力
- 当前与目标的差距
- 推荐起点（从哪个阶段/章节开始）

### 阶段路线
每个阶段必须包含：
- 阶段目标
- 学习内容（具体到知识点）
- 使用资料（具体到章节/页面/小节/repo 路径）及角色（主线/辅助）
- 练习方式与题量估计
- 掌握标准（可验证的检查点）
- 产出证据（作品、分数、能独立完成什么）
- 预计用时

### 资料清单
- 主线资料、辅助资料、可选资料
- 取舍说明（选了什么、没选什么、为什么）
- 缺资料标注（哪些内容没有覆盖到）

### 进度指针
- `当前阶段: X / Y`
- `上次完成: 阶段X, 第Y节`
- /learn-today 读到这个指针就能定位今天该学什么

### 关键 tradeoff 与风险
- 深度 vs 速度
- 理论 vs 实践
- 资料覆盖不足
- 自报起点不确定

## 4. 草案确认

- 向用户展示草案的关键决策和 tradeoff
- 用户可多轮修改
- 修改只影响草案，不影响正式计划
- 必须包含 material curation gate：基于 Phase 1 的目标资料评估和 Phase 2 的诊断结果，说明主线/辅助/候选/拒绝资料、片段范围、能力用途、用户起点适配、下载验证状态和 open risks
- 只有当用户明确确认计划草案与 material curation 后，才进入正式落盘

## 5. 正式落盘

用户确认后，生成正式产物：

- `<学习根目录>/learn-plan.md`
- `<学习根目录>/materials/index.json`

learn-plan.md 要求：
- 稳定的 Markdown section 结构，便于 /learn-today 解析
- 进度指针区块清晰可读
- "学习记录"和"测试记录"区块可继续追加
- 每阶段有明确的掌握标准和产出证据

materials/index.json 要求：
- 所有资料的条目、角色、类型、可下载性、缓存状态、segment 定位
- 能被 learn_materials.downloader 消费

## 6. 动态调整机制

当后续 /learn-today 或 /learn-test 的复盘结果显示需要微调时，走 mini approval：

**触发条件**（满足其一）：
- 连续 2 次以上同类薄弱项出现
- 用户主动提出调整需求
- 资料无法访问或不适用

**流程**：
1. 主 agent 说明需要调整什么、原因、建议方案
2. 用户确认或修改
3. 更新 learn-plan.md 相关部分（不重写整份计划）
4. 不重跑 Phase 1-3

**允许直接修改**（用户确认后）：
- 进度指针
- 某阶段的资料替换或补充
- 练习策略调整
- 阶段掌握标准微调

**不允许直接修改**：
- 整体学习目标
- 阶段顺序（除非用户主动要求）

## 7. 用户可见输出

草案阶段：
- 草案摘要（画像、阶段路线概览、关键 tradeoff）
- 需要用户确认的决策点
- 当前是草案，不是正式计划

正式落盘后：
- learn-plan.md 路径
- materials/index.json 路径
- 进度指针
- 下一步：`/learn-today`

## 8. 禁止事项

- 不要跳过草案直接落盘
- 不要在用户未确认时写正式 learn-plan.md
- 不要无差别堆砌资料而不做取舍说明
- 不要把中间阶段文档直接当正式计划
